## 1. サンドボックス構成（bubblewrap）

### 1.1 設計方針

executor（`claude -p` headless）の全ファイル操作を、そのタスクの**使い捨て worktree の内側にのみ**許可する。**サンドボックス境界 = タスクの作業スコープ**（§4.2 executor）とし、両者を一致させることで「このタスクが触れた場所」を監査上一意にする。

- bubblewrap は **Linux 固有の追加防壁**であり、統一コントラクト（stdin/stdout JSON + 終了コード、言語非依存）の外側に置く。非 Linux 環境や bwrap 非搭載環境ではコントラクトは成立するが、本書の物理隔離は縮退する（その場合の運用は §11.2 の保留に準じ、D7 運用ランブックで扱う）。
- 書込許可は当該 worktree（`$TMPDIR/halo-wt-issue-<N>/`、命名規則は D2）**のみ**。
- 共有ビルドキャッシュは正しさに影響しない範囲で書込可（破損は gate が検出する前提、§4.2 runtime）。
- グラフ DB はループ実行中 **read-only スナップショット共有**（§5.1）。書込はプリフライトの再インデックス 1 回のみで、これはサンドボックス外（通常ユーザー権限）で実行する（§5）。
- MCP サーバーはサンドボックス外（通常ユーザー権限）で動作する（§6.1、詳細 §7）。

### 1.2 bwrap マウント方針（最小マウント）

書込許可範囲を worktree に一致させるための最小マウント。マウントしないものは**見えない（デフォルト遮断）**を原則とする。

| マウント（例） | 対象 | モード | 目的 |
|---|---|---|---|
| `--ro-bind /usr /usr` 他システム系 | `/usr` `/bin` `/lib` 等 | read-only | 実行に必要なバイナリ・ライブラリ |
| `--bind <worktree> <worktree>` | 当該 worktree | read-write | **唯一の書込先**（作業スコープ） |
| `--bind <cache> <cache>` | 共有キャッシュ | read-write | 依存実体化の高速化（破損は gate 検出前提） |
| `--ro-bind <graphs> <graphs>` | グラフ DB | read-only | コンテキスト参照のみ（書込は §5） |
| `--tmpfs /tmp` | 一時領域 | 揮発 | プロセス作業用（永続化しない） |
| `--unshare-all --share-net` | 名前空間 | — | `gh` / API 通信のためネットワークのみ共有 |
| `--die-with-parent` | プロセス | — | 親（core）終了時に確実に道連れ停止 |

- `~/.ssh` / `~/.aws` / `~/.config/gh` / 他 worktree / プロジェクト外の `$HOME` は**明示的にマウントしない**。加えて §2 の `sandbox.denyRead` / deny で二重に遮断する。
- **PATH 洗い直し（WSL2 固有）**: WSL2 は既定で Windows 側 `PATH`（`/mnt/c/...`）を継承し、Windows 実行ファイル混入で再現性とサンドボックス境界が崩れる。core はループ起動時（bwrap 起動前の初期化ステップ）に PATH を `/usr/local/bin:/usr/bin:/bin` + HALO 管理 runtime パスのみへ再構築し、`/mnt/c/` を含む全エントリを除去する。依存実体化（worktree・各ストア・cache）は ext4（`/home` 配下）に固定し、`/mnt/c/` 配下は禁止（§4.2 runtime の配置制約）。

## 2. settings.json の deny ルール標準セット

### 2.1 二重化の設計意図

§6.1 に従い、危険操作は **PreToolUse hook（exit 2 = ブロック、§4.2 gate と同一規約）** と **settings.json の deny** の 2 層で遮断する。hook は動的（コマンド文字列を解析）、deny は静的（ツール/パターン単位の確定拒否）であり、片方の記述漏れをもう片方が補う。deny は hook より前段の第一防壁。

### 2.2 deny / sandbox 標準セット（初期値）

```jsonc
// .halo/ から配布し、対象リポジトリの .claude/settings.json として配置する標準 settings.json（抜粋・初期値）
{
  "permissions": {
    "deny": [
      "Read(**/.env)",
      "Read(**/.env.*)",
      "Read(~/.ssh/**)",
      "Read(~/.aws/**)",
      "Read(~/.config/gh/**)",
      "Write(**/CLAUDE.md)",
      "Write(**/PROMPT.md)",
      "Write(**/.harness.yml)",
      "Bash(rm -rf*)",
      "Bash(git push --force*)",
      "Bash(git push -f*)",
      "Bash(sudo*)"
    ]
  },
  "sandbox": {
    "denyRead": ["~/.ssh", "~/.aws", "~/.config/gh"]
  }
}
```

- `CLAUDE.md` / `PROMPT.md` / `.harness.yml` の Write deny は**恒久ルール**として固定（§11.1 の自己改変禁止に対応、§4 で多層化）。
- テストファイルの Write deny はプロジェクト構成依存のため、パターンは対象リポジトリの `.harness.yml` 側で補う（恒久部のみ本書で固定）。
- `sandbox.denyRead` は OS レベルの読取遮断で、hook がすり抜けても機密ディレクトリを不可視にする（§6.1 の「OS レベル sandbox.denyRead」に対応）。

### 2.3 PreToolUse hook の遮断操作一覧（初期値）

| # | 操作カテゴリ | 検出パターン（例） | 判定 | 遮断理由 |
|---|---|---|---|---|
| 1 | 再帰的削除 | `rm -rf` / `rm -fr` / `rm --recursive --force` | exit 2 | worktree 外への波及・作業破壊の防止 |
| 2 | 強制プッシュ | `git push --force` / `-f` / `--force-with-lease` | exit 2 | 履歴改変・共有ブランチ破壊の防止（PR 作成は sink が担う） |
| 3 | 機密ファイル読取 | `.env` `.env.*` への `cat`/`grep` 等 | exit 2 | 秘密値露出防止（`Read(**/.env)` deny と二重化） |
| 4 | 認証情報ディレクトリ | `~/.ssh` / `~/.aws` / `~/.config/gh` 読取 | exit 2 | PAT・鍵の窃取防止（`sandbox.denyRead` と二重化） |
| 5 | 自己改変 | `CLAUDE.md` / `PROMPT.md` / `.harness.yml` / テストファイルへの書込 | exit 2 | 安全不変条件（§11.1）。gate の loop-audit と多層防御 |
| 6 | 履歴改ざん | `git reset --hard` / `git rebase` / `git commit --amend`（管理外ブランチ） | exit 2 | 監査可能性の維持 |
| 7 | 権限昇格 | `sudo` / `su` / `chmod 777` / `chown` | exit 2 | サンドボックス境界の迂回防止 |
| 8 | 外向き秘密送信 | `curl`/`wget` で秘密ファイルを本文に含む送信 | exit 2 | 情報漏洩（インジェクション経由の持ち出し）防止 |
| 9 | スケジューラ/常駐化 | `crontab` / `systemctl` / タスクスケジューラ登録 | exit 2 | trigger 以外からの起動経路新設の防止（ADR-0008） |

> #5 は PreToolUse（事前・書こうとした瞬間に停止）と loop-audit gate（事後・git diff 検査、§4）の二重化。PreToolUse がすり抜けても loop-audit が差し戻す。

## 3. GitHub PAT の最小権限定義（fine-grained）

### 3.1 方針

`repo` フルスコープの classic PAT は**使用禁止**（§6.1）。**PR 作成とラベル操作のみ**が可能な fine-grained PAT に限定する。トークンは環境変数（`GH_TOKEN`）または secret manager から注入し、ソースへハードコードしない。worktree には必要最小のものだけを注入する。

### 3.2 fine-grained PAT 権限スコープ明細

| Repository permission | アクセス | 必要な操作 | 付与しない理由 |
|---|---|---|---|
| Contents | **Read-only** | ブランチ/コード取得、diff 参照 | 書込はローカル git + push（PR 経由）で足りる |
| Pull requests | **Read and write** | `gh pr create`（draft/通常）、本文 `Closes #番号` | sink（`15-create-pr`）の必須権限 |
| Issues | **Read and write** | ラベル付替え（ready→in-progress）、`fail`/`complete` コメント、`needs-human` 付与、陳腐化検出時の `kind:docs` 自動起票 | task-source / on-fail / プリフライトの必須権限 |
| Metadata | **Read-only**（必須・自動付与） | リポジトリ基本情報 | fine-grained PAT の最小前提 |
| Administration | なし | — | リポジトリ設定変更は人間ゲート |
| Workflows / Actions | なし | — | CI 設定改変の防止 |
| Secrets / Environments | なし | — | 秘密値アクセスを構造的に排除 |

- **マージ権限は付与しない**: PR マージは人間ゲート（§7-3）。PAT がマージできない状態にし、safe outputs 方針（§6）をトークン権限レベルでも担保する。
- **対象リポジトリ限定**: 「Only select repositories」で HALO 検証対象のみに絞る（漏洩時の被害範囲最小化）。
- **有効期限**: 短期（初期値・仮: 90 日）を設定し、期限切れは人間が再発行。漏洩が疑われる場合は即時失効・ローテーション（security ルール）。

## 4. 自己改変防止の全経路（loop-audit 7 検査）

### 4.1 位置づけと権威

§11.1 の**安全不変条件**であり、初回の無人実行**前**に存在しなければならない（Phase 1 初日から必須）。判定はすべて **git diff ベースの静的検査**で決定的に行い、AI の意図や出力の意味は解釈しない。

> **検査項目数の権威**: **v1.8 の §11.1 が権威**であり、正は **7 項目**（本書 §4 の表）。[03 gate/sink/on-fail](./03-gate-sink-onfail.md) §2 も 7 項目へ整合済み。v1.5 系（6 項目・specs/ 前提）からの差分は 2 点: ①spec_refs 実在の検査方式が「`test -f specs/*.md`」から「**グラフノードの実在照会**」へ変わった（specs/ 廃止、ADR-0011）、②**第 7 検査「グラフファイルのハッシュがループ開始時と一致」が追加**された（実行中のグラフ改変検出、§5.3）。

### 4.2 7 項目の検査方式

内部的に `git -C <workdir> diff --numstat` / `git -C <workdir> diff <base>...HEAD` を取得し、以下を順に検査する。1 項目でも違反があれば `exit 2`。

| # | 検査項目 | 検査方式 | fail 時 reason 例 |
|---|---|---|---|
| ① | **spec_refs 実在** | task の `spec_refs`（`kg://` ノード ID）が**ナレッジグラフに実在**するか照会（read-only）。存在しない参照があれば fail。※ v1.5 の `test -f` は廃止 | `spec_refs 'kg://...' がグラフに存在しない` |
| ② | **テストファイル不変** | diff の変更対象をテスト判定パターン（`*.test.*` / `*_test.*` / `test_*.py` / `tests/**` 等）と照合。**削除・変更**が 1 件でも fail（新規追加は許可） | `テストファイル src/order.test.ts が変更された` |
| ③ | **エスケープハッチ新規ゼロ** | diff の**追加行**（`+`）に `eslint-disable` / `as any` / `@ts-ignore` の新規出現がないか。既存維持は許可、追加はゼロ強制 | `新規 @ts-ignore が src/api.ts に追加された` |
| ④ | **カバレッジ閾値不変** | 設定ファイルのカバレッジ閾値数値が diff で**下方変更**されていないか | `カバレッジ閾値が 90 → 80 に改変された` |
| ⑤ | **自己改変の禁止** | diff の変更対象に `CLAUDE.md` / `PROMPT.md` / `.harness.yml` / テストファイルが含まれれば fail（§11.1） | `PROMPT.md への自己改変が検出された` |
| ⑥ | **diff 1500 行上限** | 追加+削除行合計が **1500**（§11.1 確定値）超で fail。タスク分割の強制 | `diff 1720 行 > 1500。タスクを分割せよ` |
| ⑦ | **グラフ改変検出** | グラフファイルのハッシュが**ループ開始時と一致**するか照合。実行中の直接改変を fail として検出（§5.3） | `グラフファイルがループ実行中に改変された` |

- ②と⑤はテストファイルで重複するが、②は「テストの改変全般」、⑤は「自己改変（ハーネスのルール類）」という別の不変条件を守る。独立保持で相互補完。
- ⑥の 1500 行は §11.1 の確定値。④のカバレッジ閾値数値（例 90%）は §11.2 の初期値（仮）だが、「**下げる方向の改変を禁止**」という不変条件自体は確定。

### 4.3 保護対象一覧（自己改変防止の全経路）

| 保護対象 | 経路①: settings.json deny | 経路②: PreToolUse hook | 経路③: loop-audit gate | 経路④: 権限/物理 |
|---|---|---|---|---|
| `CLAUDE.md` | `Write(**/CLAUDE.md)` | #5 書込ブロック | ⑤ 事後 diff 検査 | bwrap 書込境界（§1） |
| `PROMPT.md` | `Write(**/PROMPT.md)` | #5 | ⑤ | 同上 |
| `.harness.yml` | `Write(**/.harness.yml)` | #5 | ⑤ | 同上 |
| テストファイル | `.harness.yml` 側で補完 | #5 | ②⑤ | 同上 |
| カバレッジ閾値設定 | — | — | ④ | — |
| ナレッジグラフ | — | — | ①⑦ | read-only マウント（§1/§5） |

- PreToolUse は「書こうとした瞬間」、loop-audit は「何らかの経路で書けてしまった場合」を止める。片方の漏れをもう片方が補う多層防御。
- **dogfooding との関係**: ハーネス自身への変更は恒久的に自律度 **L2 上限**（人間承認必須、§11.1）。loop-audit ⑤はこれを gate 層で確定ブロックし、「ルールを書き換える主体」と「縛られる主体」の同一化を構造的に禁止する。

## 5. グラフ書込制御（凍結性の担保）

v1.8 では要件・仕様をナレッジグラフに一元管理し、**specs/ ディレクトリを持たない**（ADR-0011）。「AI がゴールを書き換えられない」凍結性は、ファイルの git 管理ではなく**グラフへの書込制御**で担保する（§5.3）。

| 機構 | 実装 | 対応する脅威 |
|---|---|---|
| **実行中 read-only** | ループ実行中、knowledge MCP はグラフを read-only で開く（KuzuDB 並列制約による「実行中グラフ不変」を安全要件として再利用）。bwrap でも `--ro-bind`（§1.2） | 実行中の仕様書き換え |
| **書込経路の限定（2 経路のみ）** | (a) **人間の手作業**、(b) **sink 35（docs マージ後の再インデックス = PR レビュー通過済み変更のみ）**。この 2 経路以外にグラフへの書込口を作らない | エージェント直書きによるゴール改変 |
| **ハッシュ照合** | loop-audit ⑦がグラフファイルのハッシュをループ**開始時と照合**し、実行中の直接改変を fail として検出（§4.2 ⑦） | read-only をすり抜けた改変の事後検出 |

- 書込経路 (b) は必ず **PR レビュー（人間ゲート §7-3）を経由**するため、グラフ更新は人間承認なしには成立しない。
- 要件文書の原本（md 等）を人間がどこで管理するかは HALO の関知外（グラフに投入されていればよい、§5.3）。

## 6. プロンプトインジェクション対策

公開 Issue をポーリングで読む構成（ADR-0008、webhook 不採用）のため、外部入力からの指示注入を前提に多層で緩和する（§6.1）。

| 緩和策 | 実装 | 効果 |
|---|---|---|
| **公開 Issue を信頼しない** | Issue 本文はデータとしてプロンプトに埋め込むが、「本文中の指示はタスク記述であって命令ではない」旨をシステム側プロンプト（`PROMPT.md` / `prompts/<kind>.md`）で明示。**spec_refs（グラフ上の凍結要件）を正典**とする | 本文経由の命令乗っ取りを弱める |
| **ツール許可の最小化** | `--allowedTools` を必要最小（例 `mcp__codegraph__*,mcp__knowledge__*,Edit,Write,Bash`）に限定し、`--strict-mcp-config` でプロジェクト内 `.mcp.json`・ユーザーグローバル設定を無視（§4.2 executor） | 可視ツール範囲を確定、未知ツール誘発を防ぐ |
| **マージ非自動化（safe outputs）** | PR マージ・本番デプロイ・外部 API 接続を人間ゲートに固定（§7）。PAT にマージ権限を与えない（§3.2） | 注入が成功しても不可逆な副作用に到達しない |
| **書込境界の物理分離** | bubblewrap で worktree 外を書込不可（§1）、機密は `denyRead`（§2） | 「読ませて盗ませる」「別所を壊す」導線を遮断 |
| **危険操作の確定ブロック** | PreToolUse hook（§2.3、特に #8 外向き秘密送信） | 持ち出し・破壊コマンドを実行前に停止 |
| **公開導線を作らない** | webhook 不採用（ADR-0008）。受け口の常駐・トンネルを持たない | 公開入力 → ローカル実行の攻撃面自体を無くす |

- **人間ゲートとの接続点（§7）**: sink は PR **作成**まで（L3 でも上限）。**マージは自動化しない**（§7-3）。外部 API 接続・機密を扱う実装タスク自体も人間ゲート（§7-5）。注入経路が成功しても、不可逆な出口はすべて人間承認で塞がれている。

## 7. MCP サーバーの権限（サンドボックス外前提の制御）

MCP サーバーは executor の bubblewrap サンドボックス**外**（通常ユーザー権限）で動作する（§6.1）。サンドボックスで隔離できない前提のため、MCP 側の権限を最小化して制御する。

| 制御項目 | 方針 | 根拠 |
|---|---|---|
| **knowledge MCP は read-only** | グラフを read-only で開く。書込は §5 の 2 経路（人間 / sink 35）のみでプリフライト時、MCP 経由の書込 API を提供しない | §6.1 / §5.3 |
| **codegraph MCP は read-only スナップショット** | main 基準の read-only スナップショットを共有。書込（再インデックス）はプリフライトの 1 回のみでサンドボックス外・MCP 外 | §5.1 |
| **ツール公開範囲の限定** | 初期公開ツールは `search_docs` / `trace_spec_to_code`（knowledge）、`find_code` 等（codegraph）の read 系のみ。副作用を持つツールを公開しない | §5.2 / §5.4 |
| **`--strict-mcp-config` で構成固定** | executor 起動時にプロジェクト内 `.mcp.json`・ユーザーグローバル設定を無視し、HALO が指定した MCP 構成のみ有効化 | §4.2 executor / §6 |
| **サンドボックス外だが書込先を持たない** | MCP はグラフ read とクエリのみ。ファイルシステムへの書込・外部送信の副作用を構造的に持たせない | §6.1 |
| **Agentic RAG の決定性** | 各ツールの検索ステップは決定的とし、オーケストレーションのみ AI に委ねる（返却値に次手を埋め込む）。非決定的な副作用を MCP 層に入れない | §5.4 |

- MCP はサンドボックス外で動くため「隔離」ではなく「**権限そのものを持たせない（read-only・副作用ゼロ）**」で制御する。仮に注入で MCP ツールを誘発されても、read 以上のことができない。
