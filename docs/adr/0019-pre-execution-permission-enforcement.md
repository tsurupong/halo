# ADR-0019: Pre-Execution Permission Enforcement (settings injection by executor)

**Date**: 2026-07-18
**Status**: proposed
**Deciders**: Owner

## Context

ADR-0004 fixes the self-modification prohibition as a safety invariant, and D4 §2/§4.3 already design a defense-in-depth of four paths (settings deny / PreToolUse hook / loop-audit gate / physical boundary). However, the current implementation enforces only the ex-post path: `gate-loop-audit` inspects the git diff **after** the executor has finished, and `executor-claude` spawns `claude -p` without injecting any settings, so the deny/hook layers designed in D4 are not wired in practice. An investigation of the Claude Agent SDK permission model (2026-07-18) confirmed that `deny` rules are evaluated before every tool call and apply **even in `bypassPermissions` mode**, making them a reliable ex-ante barrier. Relying on the ex-post gate alone means a forbidden write wastes the whole iteration (the diff is rejected wholesale) and leaves a window in which the harness rules were actually modified inside the worktree.

## Decision

Enforce the ADR-0004 protected set in two mandatory layers: **layer 1 (ex-ante)** — `executor-claude` injects a HALO-managed settings file (the D4 §2 deny standard set, including `Write(**/CLAUDE.md)` / `Write(**/PROMPT.md)` / `Write(**/.harness.yml)` and test-file patterns supplied by `.harness.yml`) into every `claude -p` invocation via `--settings`; **layer 2 (ex-post)** — `gate-loop-audit` keeps the 7-check diff inspection unchanged as the backstop. The injected settings are generated by the core at loop start from HALO-managed sources only (never read from the target worktree), so the executor cannot influence its own restrictions.

## Alternatives Considered

### Alternative 1: Keep ex-post gate only (status quo)
- **Pros**: No new wiring; single enforcement point is easy to audit.
- **Cons**: A forbidden write discards the whole iteration; the rules are briefly modifiable inside the worktree; contradicts ADR-0004's own rationale that safety invariants should be blocked ex-ante.
- **Why not**: The ex-ante mechanism is officially supported (deny survives even `bypassPermissions`) and cheap to inject; paying an iteration per violation is pure waste.

### Alternative 2: Rely on static placement of `.claude/settings.json` in the target repository
- **Pros**: No launch-time flags; matches the current D4 §2.2 wording ("distributed and placed").
- **Cons**: A file inside the worktree is itself writable state; nothing guarantees it exists or is intact when the executor starts; deny rules protecting the settings file with the settings file is circular.
- **Why not**: Injection at spawn from a HALO-managed path outside the worktree removes the executor's ability to touch its own permission source.

## Consequences

### Positive
- Forbidden writes are stopped at the moment of the tool call; iterations are no longer wasted on violations the gate would reject anyway.
- The D4 §4.3 four-path table becomes true in the implementation, not only on paper.
- Works under any permission mode, including a future `bypassPermissions` profile, because deny has top evaluation priority.

### Negative
- `executor-claude` gains a dependency on the settings-file location contract (env `HALO_SETTINGS_FILE` or equivalent, defined in D4).
- Two layers must be kept consistent: a pattern added to the gate must be mirrored into the deny set (single authoritative list stays D4 §4.3).

### Risks
- Drift between deny patterns and gate checks → mitigate by generating both from the same authoritative list (D4 §4.3) and adding a doctor check that the injected settings match it.
- `--settings` behavior changes across Claude Code versions → covered by gate-runtime-check and the ex-post gate remaining fully authoritative.
