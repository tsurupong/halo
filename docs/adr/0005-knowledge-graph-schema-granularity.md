# ADR-0005: ナレッジグラフのスキーマ粒度（ノード5種・エッジ5種で固定）

**Date**: 2026-07-09
**Status**: accepted
**Deciders**: 本人（HALO要件定義書 v1.5 §11.1 より記録）

## Context

ナレッジグラフのスキーマ粒度は後から変えるコストが高い（既投入データの再モデリングが必要）。一方でエンティティ・フィールドレベルまで下ろすと、tree-sitter で自動生成されるコードグラフと二重管理になる。

## Decision

ノードは 5 種（境界づけられたコンテキスト / 集約 / ドメイン用語 / 文書 / 決定）、エッジは 5 種（`BELONGS_TO` / `DEFINED_IN` / `IMPLEMENTED_BY` / `SUPERSEDES` / `AFFECTS`）から開始し、エンティティ・フィールドレベルには下ろさない。橋渡しエッジ `IMPLEMENTED_BY` は集約→ディレクトリパスで張る。

## Alternatives Considered

### 代替案1: フィールドレベルまで含む詳細スキーマ
- **Pros**: きめ細かいトレーサビリティ
- **Cons**: コードグラフ（自動生成）と重複し、二重管理・陳腐化の温床になる。人間設計での投入コストが跳ね上がる
- **Why not**: 詳細はコードグラフの担当領域。ナレッジグラフは人間の暗黙知（設計意図・用語・決定）に特化する

### 代替案2: スキーマレス（自由なノード/エッジ）
- **Pros**: 初期投入が楽
- **Cons**: docs-md runtime の用語集整合チェック等、グラフを参照する自動ゲートが書けない
- **Why not**: グラフを品質ゲートの基盤として使うため、型が必要

## Consequences

### Positive
- コードグラフとナレッジグラフの責務分離が明確（自動生成 vs 人間設計）
- 集約→ディレクトリパスの橋渡しにより、設計書⇔実装の対応が機械的に辿れる

### Negative
- ディレクトリ構成のリファクタで `IMPLEMENTED_BY` エッジの張り替えが必要

### Risks
- 5+5 で表現しきれない知識の出現 → エッジ追加は「開始セット」からの拡張として許容（ノード種の追加は再検討事項）
