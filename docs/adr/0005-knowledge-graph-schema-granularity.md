# ADR-0005: Knowledge Graph Schema Granularity (fixed at 5 node types, 5 edge types)

**Date**: 2026-07-09
**Status**: accepted
**Deciders**: Owner (recorded from HALO Requirements Specification v1.5 §11.1)

## Context

The granularity of the knowledge graph schema is costly to change later (it requires re-modeling already-ingested data). On the other hand, dropping down to the entity/field level causes double management with the code graph auto-generated by tree-sitter.

## Decision

Start from 5 node types (bounded context / aggregate / domain term / document / decision) and 5 edge types (`BELONGS_TO` / `DEFINED_IN` / `IMPLEMENTED_BY` / `SUPERSEDES` / `AFFECTS`), and do not drop down to the entity/field level. The bridging edge `IMPLEMENTED_BY` connects aggregate → directory path.

## Alternatives Considered

### Alternative 1: A detailed schema including the field level
- **Pros**: Fine-grained traceability.
- **Cons**: Overlaps with the (auto-generated) code graph, becoming a hotbed of double management and staleness. The cost of human-designed ingestion skyrockets.
- **Why not**: Detail is the code graph's domain. The knowledge graph specializes in human tacit knowledge (design intent, terminology, decisions).

### Alternative 2: Schemaless (free-form nodes/edges)
- **Pros**: Easy initial ingestion.
- **Cons**: Automated gates that reference the graph — such as the docs-md runtime's glossary-consistency check — cannot be written.
- **Why not**: Because the graph is used as the foundation for quality gates, types are needed.

## Consequences

### Positive
- Clear separation of responsibilities between the code graph and the knowledge graph (auto-generated vs. human-designed).
- The aggregate → directory-path bridge lets the design-doc ⇔ implementation correspondence be traced mechanically.

### Negative
- Refactoring the directory structure requires re-wiring the `IMPLEMENTED_BY` edges.

### Risks
- Emergence of knowledge that cannot be fully expressed with 5+5 → edge additions are tolerated as extensions from the "starting set" (adding node types is a matter for reconsideration).
